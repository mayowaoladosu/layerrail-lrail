# ADR-006: Kata-isolated build execution and residue quarantine

- **Status:** Accepted
- **Date:** 2026-07-13
- **Complements:** ADR-005, ADR-018, and ADR-019

## Context

BuildKit executes untrusted customer-controlled graphs. A valid Build IR and deterministic LLB definition do not prove that execution cannot reach the host, Kubernetes API, another tenant, internal metadata, control databases, or reusable credentials. Disposable workers also leave snapshots, mounts, network sandboxes, projected credentials, cgroups, and temporary controller data unless completion includes a verified cleanup protocol. Ordinary container isolation and best-effort deletion are not sufficient for this boundary.

## Decision

Each production build cell has a Go controller with no control-database route. It accepts only canonical signed assignment envelopes over TLS 1.3 mutual authentication. Ed25519 verification precedes semantic error disclosure. The verifier checks key ID, signature, cell audience, immutable resource scopes, whole-second issuance/expiry, bounded TTL, monotonic generation, nonce, complete definition-lock digest, source/output object references, and exact LLB/config identities. Artifact bytes are fetched only after verification and are re-hashed. The serialized LLB is then audited independently: only the signed local snapshot and digest-pinned base materials may be sources; host networking, insecure execution, frontends, CDI, SSH, secret environment variables, and undeclared mounts fail closed; every distinct base, network/gateway, cache, and secret authority must appear in the graph exactly as locked. A proof-carrying resolved value repeats this audit and rejects mutation at each worker handoff.

Replay and lifecycle state use cell-local transactional Bolt databases on a persistent volume. Reservation atomically retains per-build generation and cell/nonce watermarks. Exact replay returns the durable terminal result, a live foreign owner returns in-progress, an expired lease can be resumed by a new controller owner, and stale or conflicting identity fails closed. Generation-bound cancellation is persisted before signaling an active process, survives controller restart, permits immediate ownership takeover, and prevents a canceled record from later accepting false success. The build cell never connects to Rails or a control-plane database.

The broker signs build assignments at the verifier's one-hour maximum; the worker Job and ephemeral mTLS can never outlive that envelope. This execution bound is independent of the substantially shorter acquisition identities below.

The controller realizes capabilities just in time:

- It authenticates to OpenBao with a projected, audience-bound Kubernetes token, obtains a bounded token, reads only signed logical secret paths, and revokes the token before returning secret bytes to the worker lifecycle. A failed immediate revocation returns a tracked cleanup lease; the controller retries it and quarantines the run if authority cannot be proven gone. The OpenBao token TTL bounds only that already-revoked acquisition identity; it never truncates the assignment-bounded worker Job, worker mTLS, or egress-policy lifetime. Secret bytes are supplied through a BuildKit secret session, wiped from owned buffers (including redaction patterns and partial log buffers), and redacted in raw/base64/URL/hex/escaped/multiline log forms.
- Network and cache sets must exactly match both the signed definition lock and audited LLB. Every networked LLB vertex carries only the fixed loopback CONNECT proxy setting. The worker-local bridge authenticates to an owned central proxy with a short-lived certificate whose signed extension contains exact build, generation, public-domain, and private-gateway authority. The proxy resolves each domain on every CONNECT, rejects any forbidden or mixed answer, dials only the validated numeric IP, and records domain-level audit metadata before opening the socket. Private DNS answers must remain inside certificate-bound exact CIDRs and ports. Kubernetes and Cilium let workers reach only DNS for the proxy Service and the proxy TLS endpoint; site overlays add only exact private endpoint rules to the proxy. A no-network LLB run remains BuildKit `NetMode_NONE`.
- Cache records are scoped by the compiler, policy digest, signed cache namespaces, and output. A local OCI cache export is validated through its index, manifest, BuildKit cache config, and layer descriptors; all blobs are content-hashed, stored as an immutable record, and atomically selected for later workers. Real conformance imports the record on a fresh daemon. Graphs that can receive secrets do not export cross-worker cache records.
- OCI/static exports are hashed before an artifact committer moves them outside disposable scratch. ADR-020 implements this seam with immutable Harbor publication: success requires a digest-qualified OCI reference, exact source and manifest identities, verification of every referenced blob, and—only for static outputs—an immutable S3 publication manifest.

A production worker is one rootless BuildKit process in a disposable Kubernetes Job on dedicated tainted build nodes. The Pod uses `runtimeClassName: kata-qemu`, a digest-pinned worker image, a scoped private-registry pull Secret, non-root UID/GID, restricted Pod Security, seccomp, AppArmor, all capabilities dropped, read-only root, no host namespaces, no host mounts, no Docker/containerd socket, no service links, and a per-build service account with token automount disabled and no RoleBinding. Per-worker ephemeral mTLS uses a unique in-memory CA and assignment-bounded server/client certificates. EmptyDir and ephemeral-storage limits enforce bytes; the worker wrapper continuously measures all daemon state, runtime files, temporary files, sockets, and symlinks under one quota root and terminates BuildKit on either byte or inode limit. Kata's shared filesystem must preserve extended attributes (`--xattr` for virtio-fs); before creating either user namespace, the strict wrapper proves a set/list/read/remove round trip in the disposable state volume and fails closed if metadata cannot be preserved. Because `no_new_privs` correctly prevents `newuidmap`/`newgidmap` file-capability elevation, the strict worker creates an outer one-ID user namespace directly, then gives each nested OCI process an exact one-ID map from policy-required logical UID/GID `10001` to that outer ID. Exact-shape BuildKit, containerd, continuity, moby-user, rootlesskit, and runc patches collapse every nonnegative image filesystem UID/GID onto that one writable ID while preserving `10001:10001` at the build-process and final OCI boundaries; negative identities fail closed, and all other mapping shapes plus the ordinary subordinate-ID path remain unchanged. The native snapshotter's strict path omits only the redundant per-entry `lchown(0,0)` after logical root has created an already-flattened clone; unexpected parent ownership fails closed, while ordinary snapshot copying is unchanged. Curated toolchain bases preserve the exact upstream rootfs in one OCI layer so the native snapshotter does not recopy the full parent tree once per upstream layer over virtio-fs; their immutable digest and upstream source identity remain policy evidence. The runtime uses version-pinned Go 1.26.5 builds of those patched dependencies; unused Git, SSH, curl, buildctl, and CNI tools are removed.

The controller retries only classified worker loss, always on a fresh attempt/Job. Arbitrary build failures are not retried as infrastructure loss. Cancellation cancels the solve first, waits a bounded grace period, then closes the client and force-deletes the Job. Every path revokes capabilities, deletes Kubernetes resources (including pre-Job crash orphans), scrubs controller scratch, and returns a cleanup condition. The complete redacted controller-plus-worker event stream and each worker transcript are length-delimited and SHA-256 hashed into the durable result. No complete event is emitted before all immutable outputs are committed and cleanup is proven.

A privileged residue agent is restricted to dedicated build nodes and is separate from customer workers. It accepts mTLS cleanup requests for one canonical build ID, Pod UID, Pod name, and its own node. Through the node CRI socket and exact allowlisted host roots it stops/removes that Pod's containers and network sandbox, unmounts only descendants of the Pod's kubelet root, removes projected credentials/emptyDir BuildKit state, clears exact Pod cgroups, and then performs a second inspection. Kubelet teardown may race the first CRI or mount pass, so the agent repeats the complete scrub-plus-inspection protocol a small bounded number of times; only a final pass with no operation error and no finding is `clean`. It rejects symlinks, traversal, malformed CRI responses, out-of-scope resources, and unbounded walks. Any final error or remaining container, sandbox, mount, volume root, credential, snapshot-bearing emptyDir, or cgroup yields `quarantined`; the controller taints the node `NoSchedule`.

Kubernetes admission and namespace policy independently enforce Kata, digest-pinned images, no API token, no host paths, restricted security context, quotas, and default-deny networking. The trusted residue DaemonSet is isolated in a privileged system namespace and is never part of the customer worker Pod.

## Local versus production evidence

Local functional conformance may use the pinned rootless BuildKit daemon and the cluster's gVisor RuntimeClass. It proves deterministic solve execution, progress/redaction and transcript identity, cancellation, worker-loss retry, immutable export, real local-cache export/import across fresh daemons, API/host/socket denial under that environment, per-connection proxy policy and DNS-rebinding behavior, Cilium policy composition, and residue reporting. It does **not** prove Kata hardware virtualization. Production readiness requires a cluster with `/dev/kvm`, the `kata-qemu` handler, dedicated build nodes, Cilium/Kyverno/cert-manager, and the Kata conformance gate. A gVisor result must always be labeled gVisor evidence rather than Kata evidence.

## Boundaries

WP-038 owns assignment transport, replay/run journals, source/LLB materialization, BuildKit execution, JIT secret/network/cache realization, structured progress, cooperative/forced cancellation, fresh-worker retry, scratch quotas, immutable local artifact identity, worker deletion, residue proof, and quarantine.

WP-039 owns Harbor project/robot credentials, immutable registry push, digest verification, retention, and replication requests; ADR-020 records the implemented boundary. WP-040 owns SBOM, scan, provenance, signatures, and final policy. Therefore a WP-039 artifact is durable and retrievable but not an admissible release, and the broader platform must not mark a deployment successful until WP-040 durable evidence exists.

## Consequences

The execution boundary has more moving parts: mTLS rotation, dedicated Kata nodes, Cilium policy, transactional cell state, OpenBao Kubernetes auth, quota wrapper, and a privileged but tightly scoped node agent. In return, workers have no ambient platform authority; retry and replay cannot duplicate successful work silently; cancellation has a terminal cleanup proof; and cleanup failure removes the node from scheduling instead of producing false success.

## Supersession

A replacement must retain signed/audience-bound assignments, durable monotonic replay protection, no control-database route, reference-only JIT credentials, exact capability realization, hardware-isolated disposable workers by default, no worker Kubernetes/host authority, byte and inode enforcement, structured redacted progress, cooperative then forced cancellation, fresh-worker retry only for classified loss, immutable retrievable output identity, always-run cleanup, second-pass residue inspection, and quarantine on every unproven scrub.
